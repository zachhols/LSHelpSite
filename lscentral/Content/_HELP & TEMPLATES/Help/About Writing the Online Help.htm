About Writing the Help

This page contains general information about writing for the Help. A part of it is from older documents on SharePoint.

If you are new to Flare and want to start writing, head over to the Getting Started with Flarepage. Make sure to read the Writing Guidelines, it contains guidelines to the setup of the help in Flare, what kind of links should be used, about images and more.

 

Content

Before you start writing, ask yourself the following questions:

  • Who is the intended audience? For the Help, the readers will both be end users of LS Central and partners, the content should be written with that in mind. Don't spend much space explaining obvious things but when in doubt it is better to explain than not to.
  • What do the readers (partners and end-users) need to know and what are they trying to accomplish?
  • What is a main issue, what is a minor issue? Often, if you have too many details, you end up writing a lot about very little and omitting some important details of the main points. Try to keep it simple.
  • If you wonder which expression to use, ask yourself: What exactly am I explaining and what is the clearest way to tell it?

Example:

Should I say: „Issue a gift card on the POS” or should I say: „Sell a gift card on the POS“? Well, if you mean that you can issue it but do not necessarily have to sell it at this point, issue is the right expression, say so. But if you cannot issue a gift card on the POS unless you are selling it, then „sell“ is the right word.

 

Language:

  • Use Plain English. English U.S. is the LS Retail official language. Make sure the language is set so. Remember that the readers are not necessarily native English speakers.
  • Do not use UPPERCASE, bold, underscore and italic for emphasis all in the same paragraph (unless you have a very good reason), use them as advised in the Writing Guidelines
  • Use the standard fonts in the LSNav.css stylesheet and do not add other fonts.
  • Use a screenshot to explain, not to decorate. Provide a link to run the page (see Hyperlinks and Cross-References ) in the screenshot if it is from LS Nav. Do this in the text above the screenshot. Frames are recommended around screenshots unless they do not need one, instructions for adding the frame are in the Writing Guidelines. There is however one instance where it is best not to use frames, that is with a combination of two cards/windows that overlap, but in that case be sure to use white background (an empty Word page will do if you don’t have that in LS Nav) for the screenshot.
  • Be careful how you address the reader. Use direct speech for clarity. It is OK to say: „You can create ...“, but do not use „I can ...“ or „We can ...“.
  • Always use simple, visual methods if possible. If you use some of Snagit's most common methods for screenshots a drop-down list keeps disappearing. A simple trick is to use the Print Screen button on your keyboard and select the text in Snagit manually instead of using the automatic snapshot options or change your ordinary settings in Snagit.
  • Use a positive approach rather than negative. Say for instance: This is supported in versions LS Nav 2013 (version 7 and later) rather than: This is NOT supported in the version LS Nav 6.4 and earlier.
  • Use Cross-References in your topics if needed, however, make sure that the text is not overflowing with references and links that confuse and overwhelm the reader. Links in the text itself should be highly related to the topic and used sparingly. If there are other topics that relate to your topic, add a See also at the bottom of the page with Cross-References. Try to keep the most related references at the top and add more links there rather than having too many in the main text.
  • Make sure that the names of the topics are informative, when you look up the content in the master TOC it should be easy to find what you are looking for. It is not so easy to know where to look for information if you only find chapters called something like: General Features, Business Central Functionality and POS Functionality all without subsections.
  • Do not use abbreviations.
