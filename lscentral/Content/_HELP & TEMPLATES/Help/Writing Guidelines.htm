Writing Guidelines

Get Help

There are pages with help to get started using Flare and writing the Help in the _HELP & TEMPLATES folder in the Content Explorer. The folder also contains templates to copy and use whenever needed. Flare also provides Dynamic Help and pdf documents if needed. It is also helpful to browse around in the Flare Help project to get a better overview of how the help should be written and organized.

 

Topics

Start a new page for each chapter with Heading 1 in longer documents (with front page). This is valid for User Guides, some Memos, all Training Manuals and White Papers. Use common sense. If a chapter is only two lines, it probably not a seperate chapter. However an Introduction of 5-10 lines may well stand alone and not belong to another chapter. Here it is justifiable to spend/waste almost a page on air if a long/detailed document follows.

 

Main Topic

The Main Topic page contains the introduction to the topic. At first there is a short general text about the functionality, then a table with description of the sub topics and links to them (See How to: Check Transaction at POS in Statement Calculation).

 

Sub Topic

Create as many sub topics as needed for the help, link to them from the table in the Main Topic page and/or place them in the tree view below the Book for the main topic (See New Sub Topic).

Headings and Style Sheets

Use h1 for main heading on the top of the page.

h2 for other headings within the page. Use h2 for See also(bottom of page) if used.

Use stylesheet LSNav.css

 

Hyperlinks and Cross-References

Use Cross-References for referencing to other pages within the Help. It is easy to update Cross-References automatically when page names or locations are changed. Cross-References appear as links in the Help, but when a pdf is exported from the project, the cross-references contain a page number instead of a link. To update Cross-References, go to Tools in the Action menu and click Update Cross-References.

Use Hyperlinks to Run client rather than writing the whole path to a page.

Example:

Do not do this: Open the POS Terminal Card at LS Retail > Point of Sale > Design > Administration > General > Terminals.

Do this: Open the Deals page, select a line and click Edit (The link to run a page looks like this: dynamicsnav://///runpage?page=99009511)

Or: Open the Deals page in the Sales & Marketing departments menu, select a line and click Edit

 

Images

Only use images/screenshots when they add great value and make sure to keep them organized in the appropriate folders. Make sure to delete all images in folders that are not in use.

Screenshots are better in full resolution, avoid resizing them if possible.

Naming of images is important, and makes it easier to update them later on if marked properly. Try to associate all images to a page number or POS, store and terminal number.

 

Examples:

  • If the image displays a page the name is: P_pagenumber_description (P_99008905_Hyper, P is for Page)
  • If the image displays a screenshot from POS it should be something like: POS_S0001_P0001_Lookup.
  • If the image displays a slip printout it should start with: SLIP_
  • If the image displays a screenshot from the departments menu it should start with: DepartmentsMenu_
  • If the image displays a screenshot from a mobile device it should start with: M_

If you have an image that does not fit any of these namings, go to the File List (View tab > Explorer > File List) and see if there are other images in the same category and use the same naming pattern. Otherwise make sure to name the image in a way that you and others would easily know what keywords to use to find them.

Notational Conventions

Convention Meaning
bold In syntax, characters that you type exactly as shown, including commands and parameters. In text, menu names and menu commands are also bold.
Bold monospace (courier new) Commands that you must type exactly as shown to get the results being discussed.
italic Variables for which you supply a specific value. For example, Filename.ext represents any valid file name.
Initial Capitals (Filename.ext) Names of files should begin with an initial capital letter, for example, Filename.ext. Paths and folders can be uppercase, lowercase, or mixed, according to how they actually appear in a standard installation of the application or the operating system.
ALL CAPITALS Used for acronyms.
monospace (courier new) Examples of code
[ ] (square brackets) In syntax descriptions, square brackets enclose optional items. If you include the item, type only the information between them, not the square brackets themselves.
{choice1 | choice2} (braces) In syntax descriptions, braces enclose items which require a choice, such as {yes | no}. Type only one of the choices, not the braces or the dividing line.
